# 无按钮回复与飞书状态色设计

## 目标

让 Telegram 和飞书中的 AI 回复保持简洁：新消息不再附带常驻按钮，所有操作统一通过 `/screen`、`/status`、`/esc`、`/cc` 等斜杠命令完成。同时，飞书卡片通过顶部颜色直观表达 tmux 内 CLI 的运行状态。

tmux 仍是唯一执行核心。Codex 和 Claude 继续运行在原有 tmux pane 中，本次改动只影响消息展示，不替换会话、不重建 pane，也不改变 provider 事件采集方式。

## 用户可见行为

- Telegram 最终回复不再显示底部 Inline Keyboard。
- 飞书普通回复和流式回复结束后不再显示操作按钮。
- 已经发送过的旧消息仍保留兼容回调，避免旧按钮突然失效。
- 新消息的操作入口统一为斜杠命令。
- 本地文档和图片仍通过渠道原生附件接口发送，不显示服务器本地路径。

## 飞书颜色规则

| 状态 | 卡片颜色 | 含义 |
| --- | --- | --- |
| `working` | 黄色 | CLI 正在执行、思考或调用工具 |
| `waiting` | 橙色 | 等待用户输入或确认 |
| `completed` / `idle` | 绿色 | 本轮任务完成或 CLI 空闲 |
| `error` / `blocked` / `dead` | 红色 | 执行失败、受阻或进程异常 |
| `info` | 蓝色 | 普通信息提示 |
| 未知状态 | 灰色 | 无法可靠判断状态 |

CardKit 流式卡片创建时固定显示黄色；成功结束时关闭为绿色。最终回复若检测到等待状态则为橙色，检测到阻塞或进程异常则为红色。

## 渲染约束

- `ReplyEnvelope.metadata["display_state"]` 可显式覆盖终端推导状态。
- 飞书 Card JSON 2.0 页脚使用灰色小号普通文本，不再使用已废弃的 `note` 组件。
- 新渲染器忽略 `ReplyEnvelope.actions`，渠道能力声明 `supports_actions=False`，避免后续功能无意恢复常驻按钮。
- 中断仍通过 `/cc` 触发；历史消息的中断按钮仍走原有二次确认和权限检查。

## 验收标准

- Codex、Claude 在 Telegram、飞书的新增回复均无常驻按钮。
- 飞书工作中、等待、完成、错误卡片分别显示黄色、橙色、绿色、红色。
- 流式卡片从黄色开始，成功完成后变为绿色。
- 附件、长文本文件、消息编辑和 tmux 会话保持原有行为。
- 部署前后 tmux session 数量和 target 不发生变化。
